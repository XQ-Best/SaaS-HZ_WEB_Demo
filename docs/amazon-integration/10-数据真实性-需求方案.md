# Amazon 运营数据真实性 — 需求方案（V3）

> **版本**：v3.0  
> **日期**：2026-07-13  
> **状态**：待实施  
> **前置**：[07-运营爬取重写-需求方案.md](./07-运营爬取重写-需求方案.md)、[09-运营爬取重写-PRD.md](./09-运营爬取重写-PRD.md)  
> **触发**：YOTO美国账号产品 TOP20 出现全 SKU ACOS=40.8%、会话/库存/转化率为 0、利润率恒 0 — 经诊断为**合成层 fallback 造假**，非 Seller Central 真实逐行数据。

---

## 1. 背景与问题陈述

### 1.1 现状（2026-07-13 实查）

| 现象 | 根因类别 | 是否真实数据 |
|------|----------|--------------|
| 39 个 SKU 的 ACOS / TACoS 均为 **40.84%** | `allocate_account_ads_by_revenue` 将账户级 ACOS 摊到每个 SKU | **否（推算）** |
| 广告花费 = 销售额 × 40.84% | 同上，按收入比例反算 spend | **否（推算）** |
| 会话、转化率、FBA 库存全 0 | BR / 库存页解析未命中，无 fallback 填假数 | 空值（非造假，但不可用） |
| 利润率全 0% | 链路无 `profit_margin` 字段，前端默认 0 | **否（占位造假）** |
| 部分商品名截断（`ter Bass Tr`） | BR 表列映射错误 | 数据质量问题 |
| 最近 `reports` / `daily` sync 失败 | 紫鸟会话 / 任务中断 | 展示过期快照 |

审计命令：`py scripts/audit_amazon_data_coverage.py`

### 1.2 与 V2 PRD 的冲突

[09-运营爬取重写-PRD.md](./09-运营爬取重写-PRD.md) §4.1 已规定：

- 广告花费 / ACOS → **Campaign Manager（SKU 级）**
- 账户广告汇总 → **仅看板**，不得写入 SKU 行
- 利润率 → V2 **无成本源**，展示 `—`

当前 Python `product_composer.allocate_account_ads_by_revenue` 与测试 `test_allocate_ads_from_acos_only` **违背上述契约**，属于必须下架的逻辑。

## 2. 核心原则（不可妥协）

### 2.1 数据真实性宪法

| # | 原则 | 说明 |
|---|------|------|
| **R1** | **只展示可溯源字段** | 每个入库字段必须能指向具体页面解析结果或合法派生公式 |
| **R2** | **禁止账户级指标下沉到 SKU** | 账户 ACOS、账户广告花费不得写入 `amazon_product_snapshot` 单行 |
| **R3** | **禁止反算填补** | 不得在仅有 ACOS 时反推 `ad_spend`；不得在仅有 spend 时编造 sessions |
| **R4** | **空值诚实展示** | 未采集到 → DB 存空/0 + UI 显示 `—` + 可选 `data_quality` 说明 |
| **R5** | **派生仅限显式公式** | 允许：`tacos = ad_spend / revenue`（当且仅当两者均来自真实采集）；`conversion = orders / sessions`（两者均真实） |
| **R6** | **无源字段不展示** | 无成本主数据时，**隐藏**利润率列，不显示 0% |
| **R7** | **失败不 Demo** | 后端模式同步失败时禁止回退 `*Local.js` 种子数据 |

### 2.2 非目标

- 不在本阶段接入 SP-API 成本 / 利润核算
- 不用 LLM 从 DOM 猜数字填库
- 不为了「表格好看」做统计估算

## 3. 业务目标

### 3.1 一句话

**CrossHub Amazon 运营侧展示的每一个数字，要么来自 Seller Central / Advertising 后台对应页面，要么明确为空 — 绝不用账户汇总冒充 SKU 指标。**

### 3.2 量化验收（YOTO美国账号 PoC）

| 指标 | 当前 | V3 目标 |
|------|------|---------|
| SKU 行 ACOS 全相同比例 | 100%（39/39） | **0%**（除非后台确实相同） |
| 有真实 `ad_spend` 的 SKU 占比 | 0%（全为反算） | ≥ **50%** 有花费 SKU 有 Campaign Manager 来源 |
| `page_views` 有值 SKU 占比 | 0% | ≥ **80%**（BR 成功时） |
| `inventory` 有值 SKU 占比 | 0% | ≥ **80%**（库存页成功时） |
| 利润率列误导性 0% | 39 行 | **0 行**（列隐藏或全 `—`） |
| 合成 fallback 触发次数 | 每次 reports 同步 | **0** |

## 4. 用户故事

| ID | 故事 | 验收 |
|----|------|------|
| US-A01 | 作为 Boss，我看到 SKU 的 ACOS 与 Campaign Manager 该 ASIN 一致，或显示 `—` | 抽 3 个有广告 SKU 人工对账 |

## 5. 数据域与真实来源映射

### 5.1 产品 TOP20 字段（强制来源）

| UI 列 | 字段 | 唯一合法来源 | 禁止来源 |
|-------|------|-------------|----------|
| 商品名 | `product_name` | 库存目录 → BR 表 | 截断单元格、ASIN 重复 |
| ASIN | `asin` | BR / 库存 | — |
| 7日订单 | `orders_30d` | BR `units ordered` | 订单页汇总误匹配 |
| 7日销售额 | `revenue_30d` | BR `ordered product sales` | — |
| 广告花费 | `ad_spend_30d` | Campaign Manager SKU / ASIN 表 | 账户 spend、ACOS 反算 |
| ACOS | `acos` | Campaign Manager 同行 | 账户 ACOS 分摊 |
| TACoS | `tacos` | `ad_spend / revenue`（两者已真实） | 复制 `acos` |
| 转化率 | `conversion_rate` | BR `unit session %` 或 `orders/sessions` | 硬编码 0 展示 |
| 会话 | `page_views` | BR `sessions` | 商品名 fuzzy 匹配库存 |
| FBA 库存 | `inventory` | 库存页 **按 ASIN** | 名称模糊 donor |
| 利润率 | `profit_margin` | **本阶段不提供** | 默认 0 |

### 5.2 账户级指标（仅账户 Tab / 总览）

| 字段 | 来源 | 不得写入 |
|------|------|----------|
| `ad_acos_snapshot` | 首页 / 广告汇总 | `amazon_product_snapshot` |
| `ad_spend_today` | 首页 / 广告汇总 | SKU `ad_spend_30d` |
| `open_orders` | 首页 | — |

## 6. 同步策略变更

### 6.1 平台级一次爬取（对齐多平台缓存策略）

- 对外触发：`scope=full`（合并 account_health + daily + reports 页面全集）
- 缓存维度：`(tenant_id, platform=amazon)` 单 snapshot，不按 UI Tab 分 TTL
- 打开页面：**只读库**；手动刷新 / TTL 过期才 crawl

### 6.2 废弃行为

| 废弃项 | 原因 |
|--------|------|
| `allocate_account_ads_by_revenue` | 制造假 SKU ACOS |
| `refreshAmazonAllWithSync` 并行 `daily` + `reports` | 双倍浏览器、snapshot 分裂 |
| 前端 `profitMargin` 默认 0 | 占位造假 |
| `enrich_from_related_inventory` 按商品名补 sessions | 不可溯源 |

## 7. 范围

### 7.1 In Scope（V3）

- Python composer / parser 真实性改造
