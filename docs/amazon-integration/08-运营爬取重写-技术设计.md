# Amazon 运营侧数据爬取 — 技术设计（V2 重写）

> **版本**：v2.0-draft  
> **日期**：2026-07-11  
> **需求依据**：[07-运营爬取重写-需求方案.md](./07-运营爬取重写-需求方案.md)

## 1. 设计原则

1. **页面先行**：每个 Seller Central 页有唯一注册项（URL、等待、解析器、产出 schema）
2. **解析与导航分离**：`Navigator` 只负责打开/翻页/登录检测；`Parser` 只负责 DOM → dict
3. **合成后置**：多源 SKU 数据在 `Composer` 合并，禁止在解析器内跨页写库逻辑
4. **失败可诊断**：每页产出 `PageDiagnostic`，汇总进 `result_summary`
5. **渐进替换**：`report_crawler.py` 保留 `crawl_amazon()` 入口，内部委托新 pipeline

## 2. 目标架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Agent handler: amazon_sync                                              │
│   crawl_amazon(browser_id, scope)  ← 对外接口不变                        │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────────────┐
│ AmazonCrawlPipeline                                                     │
│   1. SessionContext（紫鸟 CDP、store_name、merchant_id cache）           │
│   2. ScopePlanner → List[PageTask]                                      │
│   3. for task: PageRunner.run(task) → PageResult                          │
│   4. Composer.compose(scope, results) → CrawlPayload                    │
│   5. Annotator（warning / synced_at / diagnostics）                      │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
┌───────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ page_registry │     │ parsers/*       │     │ composer/*      │
│ (URL 地图)    │     │ (纯 JS/DOM)     │     │ products/orders │
└───────────────┘     └─────────────────┘     └─────────────────┘
```

## 3. 核心类型

### 3.1 PageTask / PageResult

```python
@dataclass
class PageTask:
    key: str                    # e.g. "orders_fba_pending"
    url: str
    area: str                   # orders | products | ads | account | ops
    parser_key: str             # "orders_v3" | "br_table" | "campaign_manager"
    paginate: bool = False
    max_pages: int = 8
    default_status: str = ""    # orders: pending|canceled|shipped
    required: bool = False      # 失败是否降级 whole job

@dataclass
class PageResult:
    task: PageTask
    ok: bool
    rows: list[dict]
    duration_ms: int
    final_url: str
    warning: str = ""           # LOGIN_REQUIRED | ZERO_ROWS | TIMEOUT
    capture_path: str = ""
```

```python
{
  "synced_at": "2026-07-11 14:00:00",

  "reviews": [...],
  "coupons": [...],
  "shipments": [...],
```

## 5. 解析器设计

### 5.1 DOM 通用能力（`dom_utils.js` 片段嵌入各 EXTRACT）

```javascript
// collectRoots: 穿透 Shadow DOM
// asinFromText: \b[A-Z0-9]{10}\b，排除 G\d{9}
// pickBestTable: 表头打分（asin + sales + units）
// parseMoney / parsePercent / parseInteger
```

### 5.2 orders_v3 解析器

**输入**：orders-v3 任意子页 DOM  
**输出 schema**：

```python
{
  "order_no": "113-xxx-xxxxxxx",  # 必填，去重键
  "asin": "",
  "sku": "",
  "product_name": "",
  "quantity": 1,
  "fulfillment_type": "fba|fbm",
  "status": "pending|packed|shipped|canceled",  # URL default 可被行文本覆盖
  "amount": "12.50",
  "currency": "USD",
  "ordered_at": "",
  "ship_deadline": "",
  "buyer_region": "",
}
```

**策略**：

1. `table tbody tr` + `[role=row]` + `kat-table-row`
2. 无结构化行时：body.innerText 按 `order_no` 切块兜底
3. 禁止无 `order_no` 的行进入结果（避免汇总行）

### 5.3 br_table 解析器

沿用并强化 `EXTRACT_PERFORMANCE_TABLE_JS`：

- 多表评分选优；metric 行数最多者优先
- 滚动 `.scrollable-table` / `window.scrollBy` 触发懒加载
- `_sanitize_br_rows`：剔除「父商品汇总」「无 ASIN」「商品名=金额」行
- 兜底 `EXTRACT_BR_BODY_TEXT_JS`

### 5.4 campaign_manager 解析器

- grid/table 双通道；表头匹配 campaign / spend / acos / sales
- 账户级 summary：`EXTRACT_ADS_SUMMARY_JS` → `metrics` 的 `ad_spend_30d` / `acos` 快照
- SKU 级：campaign 行带 ASIN 则 `_merge_campaign_ads_into_products`
- 无 ASIN 的 campaign 仅计入账户 metric，不污染 SKU

### 5.5 inventory_grid 解析器

- 输出 `{asin, sku, product_name, inventory}`，无销售指标
- 作为 `Composer` 的 catalog 底表

## 6. 数据合成（Composer）

### 6.1 产品行合成顺序

```
catalog (inventory + home catalog)
  ↓ 按 asin merge，保留最长 product_name
+ br_rows (revenue, orders, page_views, conversion_rate)
  ↓ 同 asin 覆盖指标，BR 优先
+ order_aggregate (近窗口 orders count → orders_30d 补全)
  ↓ 仅当 BR orders 为空且订单 asin 有效
+ ad_campaigns (ad_spend_30d, acos)
  ↓ 同 asin 叠加；计算 tacos = ad_spend / revenue（revenue>0）
→ filter_valid_product_rows
→ _enrich_product_rows (rank_no, currency)
```

### 6.2 订单去重

- 主键：`order_no`
- 同一订单多页出现：保留 `status` 优先级 `canceled` > `shipped` > `packed` > `pending`

### 6.3 指标合并

- `home` 文本 → `parse_seller_central_text` → `metrics[]`
- 广告账户 summary 合并进 `metrics`（`ad_acos_snapshot` 等 key）

## 7. PageRunner 流程

```python
def run(ctx: SessionContext, task: PageTask) -> PageResult:
    t0 = now()
    page = ctx.page
    try:
        page.goto(task.url, wait_until="domcontentloaded", timeout=120_000)
        wait_for_page(task)  # 按 registry：固定 sleep / networkidle / selector
        if ctx.looks_login_page(page):
            return fail("LOGIN_REQUIRED", capture=ctx.screenshot(task.key))
        rows = []
        for page_no in range(task.max_pages):
            batch = parse(task.parser_key, page)
            rows.extend(batch)
            if not task.paginate or not click_next_page(page):
                break
        rows = post_process(task, rows)
        return PageResult(task, ok=True, rows=rows, duration_ms=elapsed(t0))
    except Exception as e:
        return PageResult(task, ok=False, warning=str(e), capture_path=ctx.screenshot(task.key))
```

### 7.1 等待策略（按页面）

| 页面 | 等待 |
|------|------|
| home | 5s |
| orders-v3 | 9s + 可选 next |
| br | 12s + scroll + networkidle 20s |
| campaign_manager | 16s + networkidle 20s |
| inventory | 10s |
| messages/reviews | 8～15s |

### 8.2 API 映射（`AmazonOperationalServiceImpl`）

读库时映射前端 camelCase：

| DB 字段 | API 字段 | 前端列 |
|---------|----------|--------|
| `orders_30d` | `orders7d`（展示口径） | 7日订单 |
| `revenue_30d` | `revenue7d` | 7日销售额 |
| `page_views` | `sessions7d` | 会话 |
| `ad_spend_30d` | `adSpend7d` | 广告花费 |

| `conversion_rate` | `conversionRate` | 转化率 |

> **说明**：V2 在 API 层增加 `period_days` 字段，前端后续可改文案；短期保持列名「7日」与 BR 实际区间一致即可。

### 8.3 过滤器双端一致

- Python：`filter_valid_product_rows` / `is_valid_product_row`
- Java：`AmazonProductRowFilter.isValidProductRow`
- 规则：ASIN 10 位、排除 GCID、商品名非金额/按钮文案/纯日期

## 9. 错误码与可观测性

| warning / error_code | 含义 | 处理 |
|----------------------|------|------|
| `LOGIN_REQUIRED` | 跳转登录页 | 任务 failed，提示重新登录紫鸟店铺 |
| `ZERO_ROWS` | 解析 0 行 | 可选页记 warning；required 页 failed |
| `PARTIAL_BR` | BR < catalog 20% | `product_sync_warning=PARTIAL_BR` |
| `NO_ASIN_ROWS` | 合成后 0 有效 SKU | 保持现有 warning |
| `AMAZON_ZINIAO_CONNECTION` | CDP 断开 | 重试 1 次 |

**日志**：Agent 结构化日志 `amazon.crawl page={key} rows={n} ms={d}`
